Before: participants had to triple-click, copy, and paste each multi-sentence canonical prompt, or retype it. Typos and partial pastes caused dbt Wizard to misinterpret intent. After: the skill auto-runs Step 1 on invocation. Pressing enter is the only way to advance. Any other input is refused with a single redirect line — Wizard will not go off-script. The instructor pauses the lab if a participant has a question.

Panel 1 · Step 1 — Locate the target model
User types (once, to invoke the skill)start scenario 2
dbt Wizard auto-runs:
I need to add support-ticket context to enriched orders without breaking downstream consumers. Find int_orders_enriched in this project. Show me what it currently produces, its grain, and which models depend on it downstream.
What participant sees: dbt Wizard searches the project, surfaces int_orders_enriched, shows its grain (one row per order) and the downstream models that depend on it.
Step 1 complete. Press enter to continue to Step 2.

Step 2: Describe stg_tickets. Show me the columns, their types, the grain, and how order_id joins back to int_orders_enriched.
Panel 2 · Step 2 — Discover the unused source
User presses[enter]
dbt Wizard auto-runs:
Describe stg_tickets. Show me the columns, their types, the grain, and how order_id joins back to int_orders_enriched.
What participant sees: Wizard returns the stg_tickets schema and confirms the order_id join key exists. Now the participant knows tickets can be wired in.

Step 3: Run a quick check: count rows in stg_tickets with a non-null order_id, count distinct ticket order_ids, and count how many of those order_ids match an order_id in int_orders_enriched. Tell me whether stg_tickets is one-to-one or one-to-many at the order grain.
Panel 3 · Step 3 — Validate the join
User presses[enter]
dbt Wizard auto-runs:
Run a quick check: count rows in stg_tickets, count distinct ticket order_ids, count matching order_ids in int_orders_enriched. Tell me whether stg_tickets is one-to-one or one-to-many at the order grain.
What participant sees: Wizard runs the validation. Participant learns stg_tickets is one-to-many (multiple tickets per order), so aggregation will be required before joining.
Step 3 complete. Press enter to continue to Step 4.

Step 4: Update int_orders_enriched to add ticket_count, has_open_ticket_flag, and last_ticket_status from stg_tickets. Aggregate stg_tickets to one row per order_id before joining. Use a LEFT JOIN so orders without tickets still appear.
