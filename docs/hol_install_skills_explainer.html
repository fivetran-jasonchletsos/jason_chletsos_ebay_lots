dbt Wizard HOL · Lab tooling explainer

install_skills_from_branch.py

The one-command "sync me to whatever the instructor just pushed" button for HOL participants. Pulls a chosen remote branch and refreshes every dbt Wizard skill on the participant's machine — without needing them to touch git.

TL;DR. The script fetches the latest branches from origin, prompts the participant to pick one (with main as default), checks it out, fast-forward pulls, then copies every entry from ./skills/ into ~/.dbt/wizard/skills/ — overwriting anything with the same name. dbt Wizard picks up the refreshed skills on its next invocation. No PyPI publishing, no manual file shuffling — instructors can ship a new skill set mid-lab and participants are one command away from running it.

What it does, in five steps

  1. Safety check. Confirms the script is running inside a git repo and that the working tree is clean. Refuses to switch branches if the participant has uncommitted local changes — so their lab progress isn't silently lost. Bypassable with --skip-clean-check if a participant knows what they're doing.
  2. git fetch origin. Refreshes the local view of every remote branch.
  3. Interactive branch picker. Lists all origin/* branches with main pinned at the top and marked as default. The participant types a number, a branch name, or just hits Enter for main.
  4. Checkout + fast-forward pull. If the local branch exists, git checkout <branch> then git pull --ff-only origin <branch>. If the branch is new locally, creates it via git checkout --track origin/<branch>. The --ff-only refuses non-clean merges — the right call for a lab.
  5. Install skills. Walks ./skills/ and copies every entry (file or directory) into ~/.dbt/wizard/skills/, overwriting same-named entries. Filters out .DS_Store and __pycache__ noise. Prints the list of installed skills so the participant sees what just landed.

Why it exists in HOL context

dbt Wizard reads custom "skills" from ~/.dbt/wizard/skills/. During a hands-on lab the instructor iterates — new skills, fixes, additions — and pushes branches as they go. This script is the equivalent of "pip install -U lab-content": one command, latest content, no git knowledge required. The lab_reference/lab_scripts/ path signals it's pre-staged tooling rather than part of the lab exercises themselves, so participants find it but don't trip over it.

What it doesn't do

  • Doesn't validate skill structure — bulk-copies whatever lives in ./skills/. Bad skills will fail at Wizard runtime, not install time.
  • Doesn't back up the destination. Hand-edits to ~/.dbt/wizard/skills/ get clobbered without warning.
  • One-way only — no upload, no sync back to the repo.
  • Doesn't update dbt Wizard itself. Just the skills directory.
  • Doesn't dry-run preview — it copies on the first invocation.

Operationally useful

  • Tell participants to cd into the cloned dbt_wizard_hol repo and run python3 lab_reference/lab_scripts/install_skills_from_branch.py.
  • If you're publishing a hotfix mid-lab, push to a branch (or to main) and tell the room which branch to type. Sixty seconds and everyone's synced.
  • For instructors: keep ./skills/ at the repo root tidy — every file/dir under it ships verbatim.
  • Exits clean on Ctrl+C (returns 130). Errors go to stderr with a one-line user-facing message; no Python tracebacks shown to participants.

Worth adding (small, high-leverage)

The current script is tight — 98% of what's in it is load-bearing for the workflow. The 2% that could go (custom exception class, symlink branch in the copy helper, Ctrl+C exit-code polish) isn't worth touching. The more useful angle is what's missing:

  1. --dry-run flag. Print the list of skills that would be installed, exit without copying. Saves the "wait, what's about to land?" support question. About five lines to add.
  2. Backup overwritten skills. If a participant hand-edited a skill, the current script silently clobbers it. Rename existing destinations to <name>.bak instead of rmtree — one-line change with real downside protection.
  3. Sanity-check dbt Wizard is installed. The script creates ~/.dbt/wizard/skills/ from scratch if missing, which means a participant who hasn't installed Wizard can run it, see "success," and then have nothing work. Warn (or fail) when the Wizard binary / marker file is absent.
  4. --branch <name> flag. Skip the interactive picker. Lets instructors paste a one-liner into chat — python3 install_skills_from_branch.py --branch lab-3-hotfix — instead of walking the room through the prompt.
  5. Surface what actually changed. Today's output is "Installed N skills." Distinguish new versus updated versus unchanged so participants can see whether the pull was meaningful.
